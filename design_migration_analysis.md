# UI 디자인 마이그레이션 기술 검토 보고서

현재 제작된 신규 엔터프라이즈 UI 디자인(목업 3종)을 기존 React 시스템(`tsx` 파일들)에 적용할 때, **핵심 프로그램 로직(데이터 페칭, 상태 관리, 저장 처리 등)을 수정해야 하는지**에 대한 검토 결과입니다.

## 結論 (결론)
**"프로그램의 비즈니스 로직(Logic) 및 상태(State) 코드는 단 한 줄도 건드리지 않고 100% 디자인만 교체하는 것이 가능합니다."**

현재 목업은 기존 소스 코드(`DashboardPage.tsx`, `DocCreatePage.tsx`, `DocHistoryPage.tsx`, `DocumentItemTable.tsx`)의 상태(State) 구조와 데이터 모델을 완벽하게 분석한 뒤 1:1로 매핑되도록 설계되었기 때문입니다.

---

## 1. 페이지별 호환성 검토 내역

### 📊 대시보드 (DashboardPage.tsx)
* **기존 로직**: API에서 `statistics`, `incomingData`(주간 데이터), `recentDocuments` 배열을 가져옵니다.
* **신규 디자인 매핑**:
  * **상단 KPI 카드 (4개)**: 기존 `statistics.todayCount`, `statistics.delayedCount` 등의 상태 변수를 그대로 `<div>{statistics.todayCount}</div>` 형태로 출력만 하면 됩니다.
  * **바 차트**: 기존 `incomingData.map()` 로직을 그대로 사용하되, 렌더링되는 `<div>`의 클래스명만 `.bar-fill` 등으로 변경하면 동일하게 동작합니다.
  * **긴급 처리 요망 (Task List)**: 기존에 받아오는 `recentDocuments` 데이터 중, 상태가 '미출고'이거나 '지연'인 데이터만 `filter()`로 걸러내어 출력해주면 추가 API 요청 없이 즉시 구현 가능합니다.
* **로직 변경 여부**: **없음** (추가 API 연동 불필요)

### 📝 문서 작성 (DocCreatePage.tsx)
* **기존 로직**: `useForm`을 이용한 폼 상태 관리 및 유효성 검사, 하단 고정 액션바의 `onSubmit` 함수가 존재합니다.
* **신규 디자인 매핑**:
  * **입력 폼**: HTML 태그에 적용되어 있던 인라인 스타일이나 기존 CSS 클래스명들을 신규 시스템인 `.input-control`, `.field`, `.doc-form-grid` 로 이름만 교체합니다. 
  * `onChange`, `value` 바인딩 속성은 기존 코드와 **완벽히 동일하게 유지**됩니다.
* **로직 변경 여부**: **없음**

### 📦 품목 입력 테이블 (DocumentItemTable.tsx)
* **기존 로직**: `items.map()`을 통해 각 품목 행을 렌더링하고, 복잡한 포매팅 로직(`formatIntegerInput` 등)을 거쳐 입력값을 처리합니다.
* **신규 디자인 매핑**:
  * 이 부분이 기존 코드와 가장 완벽하게 동일하게 설계되었습니다.
  * 1개 아이템당 메인 정보(`.doc-item-main-row`)와 비고란(`.doc-item-note-row`)으로 2줄씩 렌더링되는 기존 구조를 HTML 목업에 그대로 반영해두었기 때문에, 껍데기(`className`)만 갈아끼우면 됩니다. 포매터 함수 등은 일절 건드리지 않습니다.
* **로직 변경 여부**: **없음**

### 📂 발행 이력 (DocHistoryPage.tsx)
* **기존 로직**: 날짜(startDate/endDate), 검색어(keyword) 등의 필터 상태와 테이블 페이지네이션 로직이 있습니다.
* **신규 디자인 매핑**:
  * 필터 영역(`.filter-grid`)에 기존 `onChange={e => setStartDate(e.target.value)}` 와 같은 핸들러만 붙여주면 기존 로직이 그대로 동작합니다.
  * '취소된 문서'의 경우, 기존 로직에서 사용하던 `item.status === 'CANCEL'` 등의 조건식에 따라 `<tr>` 태그에 `.row-cancelled` 클래스만 추가되도록 삼항 연산자만 부여해주면 됩니다.
* **로직 변경 여부**: **없음**

## 2. 작업 방식 제안 (How to Apply)

기존 기능에 버그가 발생할 확률을 0%로 만들기 위해, 다음과 같은 방식으로 순수 View Layer(JSX 구조와 CSS)만 분리하여 작업할 수 있습니다.

1. **CSS 토큰 통합**: 신규 목업에 사용된 색상, 폰트 크기, 버튼 스타일을 전역 `globals.css`에 추가합니다.
2. **구조 치환(Markup Replacement)**: 각 `.tsx` 파일에서 기존 `<div>` 구조를 신규 목업의 HTML 구조로 교체합니다. 이때 데이터 바인딩 `{data}` 부분과 이벤트 `{onClick}` 부분은 원본 그대로 복사해서 붙여넣습니다.
3. **컴포넌트 분리 없음**: 만약 로직이 복잡하게 얽혀 있다면 새로운 리액트 컴포넌트를 만드는 대신, 현재 있는 페이지 파일 내에서 렌더링 태그들만 수정하므로 매우 안전합니다.

## 최종 의견
새로운 디자인은 겉보기에는 전면 개편된 것처럼 보이지만, 뼈대(DOM 트리와 데이터의 종류)는 기존 코드를 충실히 따르고 있습니다. 따라서 **기존의 백엔드 통신, 데이터 검증 로직 등을 단 한 글자도 수정하지 않고 HTML/CSS 마크업 치환만으로 안정적인 100% 적용이 가능**합니다. 승인해 주시면 즉시 기존 `.tsx` 파일들의 스타일 적용 작업(JSX 마이그레이션)을 진행하겠습니다.
